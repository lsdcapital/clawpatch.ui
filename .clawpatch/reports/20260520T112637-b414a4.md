## high: Managed worktree paths can collide for distinct finding IDs

id: fnd_sig-feat-library-a1562e140a-cc53_15be9fe2c5
category: data-loss
confidence: high
triage: confirmed-bug
status: open
feature: Node source src/main (feat_library_a1562e140a)

evidence:
- src/main/services/repoService.ts:372-380 (managedWorktreeForFinding)
- src/main/services/repoService.ts:842-851 (sanitizeWorktreeFragment)
- src/main/services/repoService.ts:457-474 (runFindingCommandLocked)
- src/main/services/repoService.ts:838-840 (findingCommandKey)
- test/main/repoService.test.ts:454-533

The managed worktree path and branch name are derived from a lossy sanitized finding ID, but the running-command lock is keyed by the raw finding ID. Distinct IDs such as `a/b`, `a:b`, `A B`, or long IDs sharing the first 80 sanitized characters can resolve to the same slug, path, and branch while still bypassing the per-finding lock. Two fix or revalidate commands can then operate on the same managed worktree concurrently, or a later command can reuse the wrong worktree for another finding, risking mixed patches and incorrect triage/revalidation state.

recommendation:
Make the managed worktree fragment collision-resistant, for example by appending a short hash of the original finding ID to the sanitized prefix. Also consider locking by the resolved worktree path or branch name so any future naming collision is still serialized or rejected.

test analysis:
The existing concurrency test only uses `fnd-1` and `fnd-2`, which sanitize to different slugs, so it proves independent worktrees for non-colliding IDs but does not exercise lossy sanitization collisions.

suggested regression test:
Add a RepoService test that starts fixes for two distinct finding IDs that sanitize to the same slug, such as `fnd/a` and `fnd:a`, and asserts they get different worktree paths and branch names, or that the second command is rejected before sharing a worktree.

minimum fix scope:
Update `managedWorktreeForFinding`/`sanitizeWorktreeFragment` naming and add collision-focused RepoService coverage.

repro:
Run two `repoService.runCommand(repoId, { command: "fix", findingId: "a/b" })` and `repoService.runCommand(repoId, { command: "fix", findingId: "a:b" })` calls concurrently. Both compute the same managed worktree slug `a-b`, but `runFindingCommandLocked` treats them as different locks because it uses the original IDs.

## high: Literal schemas are built from arrays instead of literal arguments

id: fnd_sig-feat-library-648a355089-277b_16b9c723a8
category: bug
confidence: medium
triage: risk
status: false-positive
feature: Node source src/shared (feat_library_648a355089)

evidence:
- src/shared/schemas.ts:4 (ClawpatchStatusSchema)
- src/shared/schemas.ts:34-35 (CommandStreamEventSchema.stream)

In Effect Schema, literal unions are normally constructed from the literal values themselves, not from a single array value. Passing the status array or stream array as one argument risks creating a schema for that array value rather than a union of allowed strings, which would reject valid payloads such as status "open" or stream "stdout" at runtime and would also distort the exported TypeScript types. This affects command request validation, GUI metadata, finding status fields, and command stream events because they all depend on these schemas.

recommendation:
Construct literal schemas with spread arguments, for example `Schema.Literals(...clawpatchStatuses)` and `Schema.Literals("stdout", "stderr")`, or use the correct Effect API for literal unions if the installed version names it differently.

test analysis:
The feature declares no linked tests, and there is no evidence of schema decode tests for status-bearing requests or command stream events.

suggested regression test:
Add tests that decode representative valid values for `ClawpatchStatusSchema`, `ClawpatchCommandRequestSchema` triage requests, `GuiMetadataSchema`, and `CommandStreamEventSchema`, and assert invalid literals are rejected.

minimum fix scope:
Update the literal schema construction sites in `src/shared/schemas.ts` and add focused schema validation tests.

repro:
Decode any value using one of these schemas, for example a triage request with { command: "triage", findingId: "f1", status: "open" } or a stream event with { runId: "r1", stream: "stdout", chunk: "x" }. If Schema.Literals treats the array as one literal argument, the decode fails even though the API contract expects those string values.
