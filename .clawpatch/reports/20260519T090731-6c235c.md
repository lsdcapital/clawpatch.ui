## medium: Concurrent repo additions can overwrite each other in repos.json

id: fnd_sig-feat-library-2985086c25-3de3_ac2d29972d
category: data-loss
confidence: medium
triage: risk
status: open
feature: Node package clawpatch-gui (feat_library_2985086c25)

evidence:

- src/main/services/repoService.ts:64-81 (readRegistry)
- src/main/services/repoService.ts:83-103 (writeRegistry)
- src/main/services/repoService.ts:165-181 (addRepo)

addRepo performs a read-modify-write cycle over the whole registry file without serialization. If two IPC calls add different repos at the same time, both can read the same initial registry, each append its own repo, and the later write will drop the earlier addition. The renderer currently disables repeated clicks, but the service and IPC handler still expose addRepo as an async operation that can be invoked concurrently.

recommendation:
Serialize registry mutations per registryPath, or implement atomic compare-and-retry writes. At minimum, make addRepo run under a mutex that covers readRegistry through writeRegistry.

test analysis:
test/main/repoService.test.ts exercises validation, serial add/list behavior, feature map reads, triage forwarding, and malformed registry fallback, but it never runs two addRepo calls concurrently or asserts that both survive in repos.json.

suggested regression test:
Add a RepoService test that uses a controllable writeRegistry or filesystem delay to run two addRepo calls concurrently, then asserts listRepos returns both repositories.

minimum fix scope:
src/main/services/repoService.ts registry mutation path plus a focused concurrent addRepo regression test.

repro:
Start with an empty repos.json, call repoService.addRepo('/repo-a') and repoService.addRepo('/repo-b') concurrently, and inspect repos.json after both complete; only one of the two additions may remain depending on write ordering.

## low: Triage note text is never persisted to GUI metadata

id: fnd_sig-feat-library-2985086c25-6729_6710956cfd
category: bug
confidence: medium
triage: risk
status: open
feature: Node package clawpatch-gui (feat_library_2985086c25)

evidence:

- src/renderer/src/components/FindingDetailPanel.tsx:17-21 (FindingDetailPanel)
- src/renderer/src/components/FindingDetailPanel.tsx:83-91 (FindingDetailPanel)
- src/main/services/clawpatchState.ts:194-215 (toFindingListItem)
- src/main/services/guiMetadata.ts:61-74 (GuiMetadataService.setNote)
- src/main/services/repoService.ts:226-233 (RepoService.setTriage)

The detail panel treats the textarea value as a local note by loading it from finding.localNote, and localNote is derived only from GUI metadata. However RepoService.setTriage only runs the clawpatch triage command and never calls GuiMetadataService.setNote, so after invalidation or reload the textarea is repopulated from unchanged metadata and the user's note disappears from the GUI. The existence of GuiMetadataService.setNote makes the intended persistence path concrete, even though the current test only asserts CLI triage forwarding.

recommendation:
In RepoService.setTriage, persist the note through GuiMetadataService.setNote after a successful triage command, or split the UI so CLI triage notes and GUI-local notes are distinct fields with explicit behavior.

test analysis:
test/main/repoService.test.ts only checks that setTriage calls the runner with the note; it does not assert that metadata.notes is updated or that a subsequent readFindingList/readFindingDetail exposes localNote.

suggested regression test:
Extend the RepoService triage test to call setTriage with a note, then getFinding or listFindings and assert localNote equals that note; also assert an empty note removes the stored note.

minimum fix scope:
src/main/services/repoService.ts setTriage behavior and a RepoService metadata persistence test.

repro:
Open a finding, enter a note, click Save triage, then reload or refetch the finding detail. The note textarea is restored from metadata.notes and will be blank unless it existed before.
