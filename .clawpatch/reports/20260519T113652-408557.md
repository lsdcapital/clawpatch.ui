## high: Literal schemas are built from arrays instead of literal arguments

id: fnd_sig-feat-library-648a355089-277b_16b9c723a8
category: bug
confidence: medium
triage: risk
status: open
feature: Node source src/shared (feat_library_648a355089)

evidence:
- src/shared/schemas.ts:4 (ClawpatchStatusSchema)
- src/shared/schemas.ts:34-35 (CommandStreamEventSchema.stream)

In Effect Schema, literal unions are normally constructed from the literal values themselves, not from a single array value. Passing the status array or stream array as one argument risks creating a schema for that array value rather than a union of allowed strings, which would reject valid payloads such as status "open" or stream "stdout" at runtime and would also distort the exported TypeScript types. This affects command request validation, GUI metadata, finding status fields, and command stream events because they all depend on these schemas.

recommendation:
Construct literal schemas with spread arguments, for example `Schema.Literals(...clawpatchStatuses)` and `Schema.Literals("stdout", "stderr")`, or use the correct Effect API for literal unions if the installed version names it differently.

test analysis:
The feature declares no linked tests, and there is no evidence of schema decode tests for status-bearing requests or command stream events.

suggested regression test:
Add tests that decode representative valid values for `ClawpatchStatusSchema`, `ClawpatchCommandRequestSchema` triage requests, `GuiMetadataSchema`, and `CommandStreamEventSchema`, and assert invalid literals are rejected.

minimum fix scope:
Update the literal schema construction sites in `src/shared/schemas.ts` and add focused schema validation tests.

repro:
Decode any value using one of these schemas, for example a triage request with { command: "triage", findingId: "f1", status: "open" } or a stream event with { runId: "r1", stream: "stdout", chunk: "x" }. If Schema.Literals treats the array as one literal argument, the decode fails even though the API contract expects those string values.

## medium: Build script skips TypeScript checking

id: fnd_sig-feat-release-51170e0f9c-f13b_e8e69a5e54
category: build-release
confidence: high
triage: risk
status: open
feature: Package script build (feat_release_51170e0f9c)

evidence:
- package.json:8 (scripts.build)
- package.json:12 (scripts.typecheck)

The reviewed feature is defined as package script build running `tsc --noEmit && electron-vite build`, but the actual `build` script only invokes `electron-vite build`. A developer, release job, or package consumer running `pnpm build` can produce release artifacts without the explicit TypeScript no-emit check that the feature promises. `check:ci` runs typecheck separately, but that does not protect the `build` entrypoint itself.

recommendation:
Change the build script to `tsc --noEmit && electron-vite build`, or make `build` delegate to the existing `typecheck` script before invoking `electron-vite build`.

test analysis:
No linked tests are included for this package script behavior, and package scripts are not validated by the listed Vitest test suite evidence.

suggested regression test:
Add a package-script assertion that `scripts.build` includes the typecheck step before `electron-vite build`, or add a CI smoke test that runs `pnpm build` against a fixture with a TypeScript error and verifies it fails before bundling.

minimum fix scope:
package.json `scripts.build`

repro:
Run `pnpm build`; it executes `electron-vite build` directly rather than `tsc --noEmit && electron-vite build`.

## medium: Completed repo commands can refresh the wrong repository after a repo switch

id: fnd_sig-feat-library-8e600da3ee-4712_ed3e1492ec
category: concurrency
confidence: medium
triage: risk
status: open
feature: Node source src/renderer (feat_library_8e600da3ee)

evidence:
- src/renderer/src/routes/ClawpatchApp.tsx:156-162 (commandMutation)
- src/renderer/src/routes/ClawpatchApp.tsx:171-186 (triageMutation)
- src/renderer/src/routes/ClawpatchApp.tsx:317-320 (RepoSidebar.onSelectRepo)

Both command and triage mutations run against the repo passed in their mutation variables, but their success handlers invalidate queries using the currently selected repo from component state. If a user starts a command for repo A and switches to repo B before it finishes, the mutation result is logged but invalidation targets repo B instead of the repo whose data actually changed. Repo A's findings, feature map, and diff can remain stale until another refresh happens.

recommendation:
Use the mutation variables passed to onSuccess to invalidate the repo that the mutation actually ran against, for example onSuccess: (result, variables) => invalidateRepo(queryClient, variables.repo.id). Apply this to both commandMutation and triageMutation.

test analysis:
No linked tests were provided for renderer mutation behavior, and this race requires simulating a repo selection change while an async mutation is in flight.

suggested regression test:
Add a ClawpatchApp test with two repos where window.clawpatch.commands.run is deferred, start a command on repo A, select repo B, resolve the command, and assert invalidation/refetch happens for repo A rather than repo B.

minimum fix scope:
Update ClawpatchApp mutation success handlers to use mutation variables for cache invalidation.

## medium: Concurrent repo additions can overwrite each other in repos.json

id: fnd_sig-feat-library-2985086c25-3de3_ac2d29972d
category: data-loss
confidence: medium
triage: risk
status: open
feature: Node package clawpatch-gui (feat_library_2985086c25)

evidence:
- src/main/services/repoService.ts:64-81 (readRegistry)
- src/main/services/repoService.ts:83-103 (writeRegistry)
- src/main/services/repoService.ts:165-181 (addRepo)

addRepo performs a read-modify-write cycle over the whole registry file without serialization. If two IPC calls add different repos at the same time, both can read the same initial registry, each append its own repo, and the later write will drop the earlier addition. The renderer currently disables repeated clicks, but the service and IPC handler still expose addRepo as an async operation that can be invoked concurrently.

recommendation:
Serialize registry mutations per registryPath, or implement atomic compare-and-retry writes. At minimum, make addRepo run under a mutex that covers readRegistry through writeRegistry.

test analysis:
test/main/repoService.test.ts exercises validation, serial add/list behavior, feature map reads, triage forwarding, and malformed registry fallback, but it never runs two addRepo calls concurrently or asserts that both survive in repos.json.

suggested regression test:
Add a RepoService test that uses a controllable writeRegistry or filesystem delay to run two addRepo calls concurrently, then asserts listRepos returns both repositories.

minimum fix scope:
src/main/services/repoService.ts registry mutation path plus a focused concurrent addRepo regression test.

repro:
Start with an empty repos.json, call repoService.addRepo('/repo-a') and repoService.addRepo('/repo-b') concurrently, and inspect repos.json after both complete; only one of the two additions may remain depending on write ordering.

## medium: Concurrent repo additions can overwrite each other in repos.json

id: fnd_sig-feat-library-a1562e140a-b3c0_221ed375d8
category: data-loss
confidence: medium
triage: risk
status: open
feature: Node source src/main (feat_library_a1562e140a)

evidence:
- src/main/services/repoService.ts:84-98 (readRegistry)
- src/main/services/repoService.ts:100-119 (writeRegistry)
- src/main/services/repoService.ts:187-202 (addRepo)

RepoService.addRepo performs an unlocked read-modify-write of the shared registry file. Two IPC invocations adding different repositories can both read the same initial registry, append independently, and then the later write replaces the earlier write, losing one added repo. Electron ipcMain.handle handlers can be invoked concurrently, so this is not only theoretical for users adding repos from multiple UI actions or windows.

recommendation:
Serialize registry mutations per appDataDir, for example by guarding addRepo with an Effect Semaphore/Mutex or a promise queue around readRegistry/writeRegistry. Keep listRepos read-only outside the mutation lock unless stronger consistency is needed.

test analysis:
test/main/repoService.test.ts covers sequential add/list behavior and malformed registry fallback, but it never runs addRepo concurrently or asserts that multiple simultaneous additions are preserved.

suggested regression test:
Add a RepoService test with two concurrent addRepo calls for distinct temp directories and a delayed write/read stub or filesystem barrier, then assert both repos are present in listRepos after both calls settle.

minimum fix scope:
src/main/services/repoService.ts registry mutation path, plus one concurrent addRepo regression test.

repro:
Start with an empty repos.json, invoke RepoService.addRepo concurrently for two valid repo directories, and arrange both calls to complete readRegistry before either writeRegistry. After both resolve, repos.json may contain only the repo from the last writer.

## low: Command output is retained and re-rendered without bounds

id: fnd_sig-feat-library-8e600da3ee-3a73_8afc77c93a
category: performance
confidence: medium
triage: risk
status: open
feature: Node source src/renderer (feat_library_8e600da3ee)

evidence:
- src/renderer/src/routes/ClawpatchApp.tsx:142-145 (commands.onStream handler)
- src/renderer/src/routes/ClawpatchApp.tsx:159-160 (commandMutation.onSuccess)
- src/renderer/src/components/CommandPanel.tsx:18-28 (CommandPanel)

Every stream chunk and command result is appended to commandLog for the lifetime of the renderer session, and CommandPanel rebuilds the full output string on every render. A verbose review/fix/status command or several long sessions can grow memory usage and make unrelated UI updates increasingly expensive.

recommendation:
Cap retained command log entries or bytes, add a clear-output action, and render output incrementally or memoize the formatted text so normal UI renders do not repeatedly format the entire history.

test analysis:
No linked tests exercise large command streams or repeated command output rendering.

suggested regression test:
Add a renderer test that sends many stream events and asserts the log is capped or trimmed according to the chosen retention policy.

minimum fix scope:
Introduce a bounded command log policy in ClawpatchApp and avoid recomputing the full formatted output on every CommandPanel render.

## low: Triage note text is never persisted to GUI metadata

id: fnd_sig-feat-library-2985086c25-6729_6710956cfd
category: bug
confidence: medium
triage: risk
status: open
feature: Node package clawpatch-gui (feat_library_2985086c25)

evidence:
- src/renderer/src/components/FindingDetailPanel.tsx:17-21 (FindingDetailPanel)
- src/renderer/src/components/FindingDetailPanel.tsx:83-91 (FindingDetailPanel)
- src/main/services/clawpatchState.ts:194-215 (toFindingListItem)
- src/main/services/guiMetadata.ts:61-74 (GuiMetadataService.setNote)
- src/main/services/repoService.ts:226-233 (RepoService.setTriage)

The detail panel treats the textarea value as a local note by loading it from finding.localNote, and localNote is derived only from GUI metadata. However RepoService.setTriage only runs the clawpatch triage command and never calls GuiMetadataService.setNote, so after invalidation or reload the textarea is repopulated from unchanged metadata and the user's note disappears from the GUI. The existence of GuiMetadataService.setNote makes the intended persistence path concrete, even though the current test only asserts CLI triage forwarding.

recommendation:
In RepoService.setTriage, persist the note through GuiMetadataService.setNote after a successful triage command, or split the UI so CLI triage notes and GUI-local notes are distinct fields with explicit behavior.

test analysis:
test/main/repoService.test.ts only checks that setTriage calls the runner with the note; it does not assert that metadata.notes is updated or that a subsequent readFindingList/readFindingDetail exposes localNote.

suggested regression test:
Extend the RepoService triage test to call setTriage with a note, then getFinding or listFindings and assert localNote equals that note; also assert an empty note removes the stored note.

minimum fix scope:
src/main/services/repoService.ts setTriage behavior and a RepoService metadata persistence test.

repro:
Open a finding, enter a note, click Save triage, then reload or refetch the finding detail. The note textarea is restored from metadata.notes and will be blank unless it existed before.
